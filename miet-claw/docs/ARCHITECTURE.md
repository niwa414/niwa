# 多尺度材料仿真 agent 架构说明

## 1. 产品边界

这个系统应当被定义为：

**面向自有 MD / KMC 软件体系的多尺度材料模拟运营与编排 agent**。

它不是通用聊天代理，核心职责是：

- 受理结构化仿真任务
- 选择 `MD-only / KMC-only / MD->KMC` 三种执行路径
- 对接 LAMMPS 和当前 KMC 引擎
- 跟踪状态、处理失败、保留产物、输出结论说明

## 2. 三层结构

### A. 产品层（你自己的品牌和前端）

负责：
- 登录、项目空间、任务发起
- 品牌化 UI
- 结果浏览、审计、归档下载
- 人机解释界面

### B. OpenClaw 控制平面

负责：
- 会话与任务入口
- 技能/工具编排
- 调用你的专用工具
- 权限控制、自动化、触发器

### C. 计算执行层

负责真正计算：
- LAMMPS（当前 MD）
- MISA-KMC / CrystalKMC（当前 KMC）
- 后处理脚本
- 结果解析器与归档器

## 3. 固定链路

### 路径 1：MD-only
输入任务 -> 生成 MD 输入 -> 运行 LAMMPS -> 解析结果 -> 归档 -> 输出解释

### 路径 2：KMC-only
输入任务 -> 生成 KMC 输入 -> 运行 MISA-KMC -> 解析长时演化 -> 归档 -> 输出解释

### 路径 3：MD -> KMC 固定链路
输入任务
-> 运行 MD/NEB 或势垒抽取
-> 生成迁移能垒数据集
-> 编译事件表/速率表
-> 生成 KMC 输入
-> 运行 KMC 长时演化
-> 汇总并解释输出

## 4. 关键对象

### 任务规范（Job Spec）
必须显式描述：
- 任务模式
- 材料体系
- 温度、晶格、元素、势函数
- MD 运行方式
- KMC 运行方式
- 归档与输出要求

### 状态机（State Machine）
每个任务的每个步骤必须有状态：
- `pending`
- `running`
- `completed`
- `failed`
- `skipped`

### 产物（Artifacts）
每个阶段产物都要显式登记，例如：
- `barriers.json`
- `event_table.csv`
- `generated_kmc.in`
- `log.spparks`
- `diffusion.csv`
- `summary.md`

## 5. 当前 KMC 软件对接现实

当前文件夹中的 KMC 源码已经不是纯黑盒：

- 文档显示支持 `plugin diffusion`
- 支持 `config_plugin diffusion ...`
- 支持 `replace cluster.xyz`
- 支持 `barrier Fe 0.65` 这种按元素注入能垒
- 支持 `nbody eam/alloy` 和 `meam`
- 源码里还能看到内嵌 LAMMPS 能量相关逻辑

这意味着第一版产品不必等“完美事件引擎”才开始。
你完全可以：

1. 先由 MD 阶段输出结构化 barrier 数据
2. 生成一个语义化事件表供审计和解释
3. 再把它压缩映射为当前 KMC 可接受的 `barrier` 指令

## 6. 为什么要分“事件表”和“KMC输入”两层

因为两者职责不同：

- **事件表**：面向科研解释、审计、追踪
- **KMC输入**：面向当前引擎兼容执行

这样未来如果你换 KMC 引擎，只改适配层，不改上层任务模型。

## 7. 失败恢复策略

### MD 失败
- 保留输入、stdout、stderr、工作目录快照
- 标记在 `state.json`
- 允许重跑 MD，不重跑已完成准备步骤

### 事件表编译失败
- 多半是 barrier 数据格式不一致
- 直接在编译步骤失败，不进入 KMC

### KMC 失败
- 保留 `generated_kmc.in`、日志和已输出 dump
- 允许从 KMC 阶段恢复

## 8. 可解释输出应包含什么

最终用户不应该只看到“跑完了”。
应该看到：

- 任务目标
- 使用的软件与版本/路径
- MD 得到了哪些迁移能垒
- 这些能垒如何转成事件速率
- KMC 在什么温度和配置下运行
- 产生了哪些主要输出文件
- 如果失败，失败在第几步、如何恢复

## 9. MVP 的正确切法

第一版不要追求“所有材料体系都能自动推理”。
先只做：

- vacancy diffusion 场景
- 固定材料体系模板
- 固定 NEB/barrier 输出格式
- 固定 KMC 输入生成规则
- 固定归档结构

把这条链做稳后，再扩成更通用体系。

## 10. 新增的自主脚本层

为了让系统不只是“执行固定 spec”，现在在执行层上面增加了一层 **自主脚本层**。

这一层负责：
- 自然语言任务理解
- 模板选择
- 生成 job spec
- 生成 MD / KMC 草稿输入
- 先做 dry-run 再决定是否真跑

它应当被理解成：

- 上面接 OpenClaw 和前端
- 下面接确定性执行层

也就是：

- **自主层负责起草和自检**
- **执行层负责真跑和归档**

这能让 mietclaw 从“只能执行别人写好的任务”升级成“能先帮你搭出一个可运行任务骨架”的专用 agent。
